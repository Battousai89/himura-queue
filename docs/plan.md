# План разработки высокопроизводительного сервиса очередей на Go

## Архитектурные решения (на основе опросника)

| Компонент | Реализация |
|-----------|------------|
| **Протокол** | Бинарный TCP-протокол с фреймами (заголовок + тело) |
| **Тип очереди** | Гибридная: in‑memory с периодическим снапшотом на диск |
| **Гарантии доставки** | Exactly‑once через sequence IDs + подтверждения + дедупликация (in‑memory map с TTL) |
| **Приоритеты** | Многоуровневая приоритетная очередь на основе `container/heap` |
| **Отложенные сообщения** | Отдельная очередь с таймерами (`time.Timer` + `container/heap`) |
| **Шардирование** | Consistent hashing с виртуальными узлами (распределение по ключу очереди) |
| **Сериализация** | Собственный бинарный формат (`encoding/binary`) |
| **Пул воркеров** | Динамический пул горутин, масштабируемый под нагрузку |
| **Логирование** | Текстовые логи (`log` или структурированный вывод в stdout) |
| **Управление** | CLI утилита на базе `flag` + отдельный HTTP‑эндпоинт для health‑checks |
| **Мониторинг** | Health‑check эндпоинт `/health` + логирование критических событий |

## Ключевые компоненты

### 1. TCP‑сервер
- Неблокирующий `net.Listener` с пулом соединений
- Каждое соединение обрабатывается в отдельной горутине
- Бинарный фрейминг: длина (4 байта) + тип (1 байт) + данные
- Команды: PUSH, POP, ACK, SUBSCRIBE, STATUS

### 2. Очередь (in‑memory)
- Приоритетная очередь на `container/heap`
- Отложенные сообщения: отдельная куча с таймерами
- Шардирование: consistent hashing по имени очереди
- Синхронизация: `sync.RWMutex` на уровень шарда

### 3. Exactly‑once гарантии
- Каждому сообщению присваивается уникальный ID (монотонный счётчик)
- Клиент подтверждает обработку командой ACK
- Сервер хранит карту подтверждённых ID с TTL (очистка по таймеру)
- Повторные сообщения с тем же ID игнорируются

### 4. Persistence
- Периодический снапшот всей памяти в бинарный файл
- Write‑ahead log для durability (опционально)
- Восстановление состояния из снапшота при старте

### 5. Динамический пул воркеров
- Канал задач с буфером
- Горутины создаются при заполнении буфера
- Простаивающие воркеры завершаются через таймаут

### 6. CLI утилита
- Подкоманды: `push`, `pop`, `stats`, `health`
- Флаги: `--host`, `--port`, `--queue`, `--priority`

### 7. Тестирование
- Unit‑тесты для всех компонентов
- Интеграционные тесты с поднятым сервером
- Бенчмарки: пропускная способность, задержки, нагрузка
